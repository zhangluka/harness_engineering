# Harness Engineering？

讲真，最近 AI 圈子里最火的话题是啥？

有人说是 GPT-5，有人说是 Claude 4.6，但有一说一，这些模型更新虽然热闹，说到底也就是参数更多、上下文更长——真正让业内人士坐不住的，是 **Harness Engineering** 这个概念。

翻译成人话就是：怎么让 AI 智能体替我们程序员干活，而且干得漂亮。

有意思的是，OpenAI 和 Anthropic 这两大巨头，前后脚发布了各自的 Harness 实践指南。OpenAI 在 2026 年 2 月扔出一篇，Anthropic 3 月紧跟着也发了一篇。表面上看，都是在聊「怎么用好 AI 智能体」，但仔细一品你会发现——这俩的思路，简直就是两条不同的路。

今天咱们就来扒一扒，这两个顶流玩家到底在想啥，以及我们能从中学到啥。

---

## 01. 核心隐喻：掌舵 vs GAN

先说最本质的区别——两家对 Harness 的隐喻就完全不同。

**OpenAI 的思路很简单：人类掌舵，智能体执行。**

这话听起来有点鸡汤，但翻译成人话就是：工程师别亲自写代码了，去设计环境、明确意图、构建反馈回路。说的更直白点，你的产出不再是代码，而是**约束系统**——AGENTS.md、规则、linter、文档。

**Anthropic 呢？玩的是 GAN 那一套。**

就是生成器-判别器那一套经典架构。搞了个三智能体系统：Planner 负责规划，Generator 负责干活，Evaluator 负责挑刺。是不是听起来有点眼熟？对，就是 AI 圈训练图像生成模型的老办法。

但话说回来，这两种思路没有谁对谁错，更多是 **「约束驱动」** vs **「架构驱动」** 的区别。

---

## 02. 架构设计：简约派 vs 豪华派

### OpenAI 的做法

OpenAI 倾向于**极简 harness**。

简单来说，就是给智能体一张「地图」，而不是一本 1000 页的说明书。AGENTS.md 控制在 100 行左右，指向更深层的文档。智能体从一个小入口进来，然后被告知「接下来该看什么」。

人类在环主要是审核 PR、给反馈。慢慢才过渡到 Agent 对 Agent 的审核。整个路径是：**人工 → 规则 → 自动化**。

### Anthropic 的做法

Anthropic 上来就加了**豪华架构**。

三智能体各司其职还不够，还搞了个「Sprint 合约」机制——每次迭代之前，Generator 和 Evaluator 先坐下来谈判：啥叫「干完」？标准是什么？谈拢了才开始干活。

然后就是一轮又一轮的迭代。Anthropic 的数据显示，用这种方式跑一个完整的游戏编辑器，8 轮迭代下来花了 6 小时、200 美金。讲真，这成本确实不低。

但代价是什么呢？**质量确实碾压**。单智能体跑只要 20 分钟、9 美金，但做出来的游戏——核心功能是坏的。

---

## 03. 上下文管理：渐进式披露 vs Context Reset

这是两家分歧最大的地方。

### OpenAI 的方案

**渐进式披露**。

说难听点，就是别给智能体塞太多东西。一开始只给入口，慢慢引导它去更深的地方找答案。文档要结构化，要编入索引，要定期清理烂文档。专门有个「doc-gardening」智能体负责扫描过时的文档并发起修复 PR。

### Anthropic 的方案

**Context Reset**。

有一说一，这个概念我第一次看到的时候也被震住了。简单解释一下：模型跑久了，上下文窗口会塞满，逻辑开始飘，严重的还会出现「Context Anxiety」——模型觉得自己快到极限了，开始提前收尾，草草结束任务。

Anthropic 的解法是：直接清空上下文，重置到一个干净的状态。但这就需要一份结构化的「交接文档」，让下一个智能体能接上之前的工作。

这两种方案哪种更好？说实话，**没有标准答案**。渐进式披露更轻量，但需要很强的文档基建。Context Reset 更暴力，但需要好的交接机制。

---

## 04. 评价体系：外部审核 vs 独立 Evaluator

这可能是最关键的差异。

### OpenAI 依赖外部

主要靠人类审核 PR，再加上其他 Agent 的审核反馈。审核通过，合并；不通过，打回重改。这就是传说中的「Ralph Wiggum 循环」——智能体自己改、自己审、循环往复，直到没人挑得出毛病。

### Anthropic 搞了个独立 Evaluator

这是 Anthropic 最有意思的创新。把「干活的人」和「挑刺的人」分开。Generator 只管写代码，Evaluator 只管找问题。

为什么不能让自己评价自己？**因为 AI 有一个迷之自信的毛病——它永远觉得自己写得特别好。**

Anthropic 通过大量调试发现：独立 Evaluator 比「自己评价自己」靠谱太多了。而且 Evaluator 是可以调的——给它足够多的例子，让它知道啥叫「好」，它就能逐步学会你挑剔的品味。

---

## 05. 模型适配：隐式 vs 显式

**这是 Anthropic 特别有意思的一个洞察。**

他们发现：随着模型能力变强，很多之前必需的 harness 组件就可以扔掉了。

比如 Opus 4.5 需要 Context Reset 来解决 Context Anxiety，但 Opus 4.6 发布后，这个功能基本可以滚蛋了——模型本身就不会再犯这个病。

再比如，原来需要把任务拆成很多个 Sprint 来跑，后来发现模型自己就能 handle 整块任务，sprint 机制也可以简化。

换句话说：**Harness 不是一成不变的，要跟着模型能力动态调整。**

OpenAI 那篇虽然没明说这个观点，但字里行间也在暗示——他们也在不断迭代自己的约束系统。

---

## 06. 共同点：英雄所见略同

说了这么多差异，但这两家其实也有不少共识：

**文档是基础设施**——两家都强调结构化文档的重要性。别想着一股脑把所有信息塞给智能体，得有条理、有层级。

**反馈循环不可或缺**——没有闭环的 agent 走不远。不管是 OpenAI 的 Ralph Wiggum 循环，还是 Anthropic 的 GAN 迭代，核心都是「改到满意为止」。

**模型会进化**——Harness 需要随模型能力调整。Anthropic 明确提出了这个观点，OpenAI 也在用脚投票。

**成本意识**——两者都关注 token 成本与运行时间。毕竟 harness 搭起来是要花钱的，不能无脑堆复杂度。

---

## 07. 成本意识

两组数据很说明问题：

| 对比项 | 单智能体 | 完整 Harness |
|--------|----------|-------------|
| 成本 | $9 | $200 |
| 时间 | 20 分钟 | 6 小时 |
| 质量 | 核心功能是坏的 | 可运行 |

**但话说回来**，如果你只是做个简单 demo，那确实没必要上豪华套餐。Harness 的复杂度应该和任务难度匹配。

---

## 08. 我们的总结

扒了这么多，两家到底谁更牛逼？

我的答案是：**都很牛逼，只是路数不同。**

| | OpenAI 路径 | Anthropic 路径 |
|--|-------------|----------------|
| **哲学** | 约束驱动，越少越好 | 架构驱动，主动加复杂度 |
| **复杂度** | 自下而上，从小规则累积 | 自上而下，从架构设计入手 |
| **评价** | 依赖外部（人工/其他 Agent） | 内建独立 Evaluator |
| **任务粒度** | 小任务快速交付 | 大任务多轮迭代 |
| **模型适配** | 隐式 | 显式强调 |

### 落地启示

1. **从小处着手** —— 先用简单 harness 验证，再逐步加架构。别一上来就搞三智能体，先跑通再说。

2. **评价先行** —— 尽早建立可量化的质量标准。什么是「干完」？什么是「干得好」？先把标准定出来。

3. **模型适配** —— 定期评估 harness 复杂度是否匹配当前模型能力。模型在进化，你的 harness 也该进化。

4. **成本意识** —— harness 成本可能远超单次运行。花 200 美金做出来的东西，和花 9 美金做出来的，可能根本不是一个东西。

5. **渐进披露** —— 不要把所有规则塞进 AGENTS.md。给智能体一张地图，而不是一本手册。

---

## 09. 核心矛盾

说了这么多，两篇文章暴露了一个**核心张力**：

> **OpenAI 倾向于 minimal harness：越少越好，靠模型能力。**
>
> **Anthropic 倾向于 structured harness：架构化设计，主动增加复杂度。**

这个矛盾没有绝对答案。取决于：
- 任务复杂度
- 模型能力
- 成本约束
- 质量要求

但正是这种**路径差异**，让整个行业有了更多探索的可能性。指不定哪天，两种思路就融合了，变成一种更优雅的方案。

**毕竟，只要还在折腾，就说明故事还没讲完。**
