# Spec-Driven Development 工具演进路线图

> 当前工具体系围绕 SDD 展开，本文档规划能力补足方向。

---

## 现有能力

| 工具 | 定位 | 状态 |
|------|------|------|
| OpenSpec | Spec 生成与 Brownfield 开发 | ✅ 已内化 |
| Ralph Loop | PhSpec 变更自动应用 | ✅ 已内化 |

---

## 待补足能力

### 1. 验证层：Spec 守护工具

已有工具能生成和应用 Spec，但需要**反向验证**能力。

| 能力 | 说明 | 优先级 |
|------|------|--------|
| **Spec Linter** | 检查 Spec 格式、语义一致性（如 OpenAPI Schema 验证） | 🔴 高 |
| **实现-Spec 对齐检测** | 运行时验证代码是否偏离原定 Spec | 🟡 中 |
| **回归测试生成** | 基于 Spec 自动生成测试用例 | 🟡 中 |

---

### 2. 追踪层：变更溯源工具

Ralph Loop 支持变更应用，但缺少**全链路追踪**。

| 能力 | 说明 | 优先级 |
|------|------|--------|
| **变更血缘图** | 追踪 Spec → Code → Deploy 的完整链路 | 🟡 中 |
| **影响范围分析** | Spec 变更会波及哪些模块/服务 | 🟢 低 |
| **Rollback 辅助** | 自动生成回滚计划 | 🟢 低 |

---

### 3. 质量门：静态检查集成

工具链需要与质量保障联动。

| 能力 | 说明 | 优先级 |
|------|------|--------|
| **Pre-commit Spec 验证** | 提交前强制检查 Spec 有效性 | 🔴 高 |
| **Diff 分析工具** | 智能对比 Spec 版本差异，输出变更摘要 | 🟡 中 |
| **依赖冲突检测** | 多个 Spec 变更时的依赖分析 | 🟡 中 |

---

### 4. 可视化层：Dashboard

需要一个**全局视图**。

| 能力 | 说明 | 优先级 |
|------|------|--------|
| **Spec 健康度看板** | 各服务/模块 Spec 覆盖率 | 🟡 中 |
| **变更流转状态** | 待审核 → 待执行 → 已执行 → 已回滚 | 🟡 中 |
| **热力图** | 频繁变更区域识别 | 🟢 低 |

---

### 5. 协作层：Review 流程

当前工具偏执行，需要增强**人机协作**。

| 能力 | 说明 | 优先级 |
|------|------|--------|
| **Diff Review UI** | Web 端可视化 Spec 变更对比 | 🟡 中 |
| **Approve Gateway** | 关键变更需要人工审批 | 🟢 低 |
| **Comment/Annotation** | 在 Spec 变更上添加讨论记录 | 🟢 低 |

---

## 实施优先级

### Phase 1: 验证层（立即）
- Spec Linter
- Pre-commit Spec 验证

### Phase 2: 可视化与追踪（中期）
- Spec 健康度看板
- 变更血缘图
- Diff 分析工具

### Phase 3: 协作增强（远期）
- Diff Review UI
- Approve Gateway
- Comment/Annotation
